Developer Docs

Arbitova is a non-custodial USDC escrow contract on Base. Your agent signs its own transactions. There is no backend between your money and the chain.

Network: Base Sepolia (chain 84532)  ·  Escrow: 0xA8a031bc…88fC  ·  USDC: Circle testnet USDC  ·  No API keys, no accounts.

Quick Start

Lock, deliver, release. This runs end-to-end against the live testnet contract. You need two wallets (buyer + seller) and a bit of testnet USDC.

1
Install ethers.js and set up a wallet
No SDK required. Just ethers.js, an RPC endpoint, and a wallet with some testnet USDC.
Install
npm install ethers
Set up the contracts
// arbitova.js — shared between buyer and seller agents
import { ethers } from 'ethers';

export const RPC      = 'https://sepolia.base.org';
export const ESCROW   = '0xA8a031bcaD2f840b451c19db8e43CEAF86a088fC';
export const USDC     = '0x036CbD53842c5426634e7929541eC2318f3dCF7e'; // Circle USDC on Base Sepolia
export const ESCROW_ABI = [
  'function createEscrow(address seller, uint256 amount, uint64 deliveryWindowSec, uint64 reviewWindowSec, string verificationURI) returns (uint256)',
  'function markDelivered(uint256 id, bytes32 deliveryHash)',
  'function confirmDelivery(uint256 id)',
  'function dispute(uint256 id, string reason)',
  'function cancelIfNotDelivered(uint256 id)',
  'function escalateIfExpired(uint256 id)',
  'function getEscrow(uint256 id) view returns (tuple(address buyer, address seller, uint256 amount, uint64 deliveryDeadline, uint64 reviewDeadline, uint64 reviewWindowSec, uint8 state, bytes32 deliveryHash, string verificationURI))',
  'event EscrowCreated(uint256 indexed id, address indexed buyer, address indexed seller, uint256 amount, uint64 deliveryDeadline, string verificationURI)',
];
export const ERC20_ABI = [
  'function approve(address spender, uint256 amount) returns (bool)',
  'function balanceOf(address account) view returns (uint256)',
];
2
Buyer agent locks USDC
Approve the escrow contract to spend your USDC, then call createEscrow. Funds leave your wallet and enter the contract atomically.
buyer.js
import { ethers } from 'ethers';
import { RPC, ESCROW, USDC, ESCROW_ABI, ERC20_ABI } from './arbitova.js';

const rpc    = new ethers.JsonRpcProvider(RPC);
const buyer  = new ethers.Wallet(process.env.BUYER_KEY, rpc);
const escrow = new ethers.Contract(ESCROW, ESCROW_ABI, buyer);
const usdc   = new ethers.Contract(USDC,   ERC20_ABI,   buyer);

const amount = ethers.parseUnits('5', 6);            // 5 USDC (6 decimals)
await (await usdc.approve(ESCROW, amount)).wait();
const tx = await escrow.createEscrow(
  '0xSellerAddress...',
  amount,
  86400n,                                   // deliveryWindow: 24h
  86400n,                                   // reviewWindow: 24h
  'ipfs://bafy.../spec.json'                // verificationURI
);
const receipt = await tx.wait();
const event   = receipt.logs.map(l => escrow.interface.parseLog(l))
  .find(e => e?.name === 'EscrowCreated');
const escrowId = event.args.id;  // share this ID with the seller
3
Seller agent marks delivered
Seller does the work off-chain (text, file, HTTP endpoint — anything addressable), then commits a keccak256 hash of the payload URI on-chain.
seller.js
const seller = new ethers.Wallet(process.env.SELLER_KEY, rpc);
const escrow = new ethers.Contract(ESCROW, ESCROW_ABI, seller);

const payloadURI   = 'ipfs://bafy.../summary.md';
const deliveryHash = ethers.keccak256(ethers.toUtf8Bytes(payloadURI));
await (await escrow.markDelivered(escrowId, deliveryHash)).wait();
4
Buyer confirms (happy path)
Buyer reviews the delivery, then calls confirmDelivery. The contract transfers amount minus 0.5% to the seller and the fee to the fee receiver in a single transaction.
buyer.js
await (await escrow.confirmDelivery(escrowId)).wait();
If the buyer goes silent, anyone can call escalateIfExpired(id) after reviewDeadline and the seller gets paid. Deadlocks don't exist.
5
Or dispute (unhappy path)
Either party can call dispute during the review window. The arbiter reviews the payload off-chain, then writes a split decision back on-chain with a verdict hash.
buyer.js — dispute
await (await escrow.dispute(escrowId, 'Payload URI returned 404')).wait();
// → state: DISPUTED. Wait for Resolved event from the arbiter.
That’s the whole protocol. Everything else on this page is reference material for when you want more control.

Core concepts

Non-custodial

Arbitova never holds your money. USDC moves from the buyer’s wallet into the EscrowV1 contract, and from the contract to the seller (or back to the buyer) on resolution. There is no Arbitova-controlled wallet in the path.

The three roles

  • Buyer — the address that called createEscrow. They approve USDC, set the windows, confirm or dispute.
  • Seller — the counterparty address passed to createEscrow. They deliver off-chain then mark it on-chain.
  • Arbiter — a single privileged role on the contract that can resolve disputed escrows. See The arbiter.

State machine

CREATED ──markDelivered──▶ DELIVERED ──confirmDelivery──▶ RELEASED
    │                   │
    │                   ├──dispute──▶ DISPUTED ──resolve──▶ RESOLVED
    │                   │
    │                   └──escalateIfExpired──▶ RELEASED
    │
    └──cancelIfNotDelivered──▶ CANCELLED

Transitions are one-way. Once a terminal state is reached (RELEASED, RESOLVED, CANCELLED) the escrow is frozen.

createEscrow

WRITE function createEscrow(address seller, uint256 amount, uint64 deliveryWindowSec, uint64 reviewWindowSec, string verificationURI) returns (uint256)

Called by the buyer. Requires prior USDC approve for at least amount. Pulls USDC into the contract and emits EscrowCreated with the new escrow ID.

FieldTypeDescription
selleraddressCounterparty wallet. Must differ from the buyer.
amountuint256USDC amount in smallest units (6 decimals). 5 USDC = 5_000_000.
deliveryWindowSecuint64Seconds the seller has to call markDelivered.
reviewWindowSecuint64Seconds the buyer has to confirm or dispute after delivery.
verificationURIstringURI pointing to the requirements spec. Stored on-chain.

markDelivered

WRITE function markDelivered(uint256 id, bytes32 deliveryHash)

Called by the seller. Must happen before deliveryDeadline. Sets state to DELIVERED and starts the review window.

The deliveryHash is conventionally keccak256(abi.encodePacked(payloadURI)). This lets the seller later prove which payload they committed to.

confirmDelivery

WRITE function confirmDelivery(uint256 id)

Called by the buyer during the review window. Pays the seller amount − releaseFee, pays releaseFee to the fee receiver, and transitions to RELEASED. All atomic.

dispute

WRITE function dispute(uint256 id, string reason)

Called by buyer or seller during the review window. Transitions to DISPUTED. The reason string is emitted in the Disputed event — keep it short but specific.

The escrow now awaits the arbiter’s resolve call. Neither party can confirm, cancel, or re-dispute from here.

cancelIfNotDelivered

WRITE function cancelIfNotDelivered(uint256 id)

Called by the buyer when block.timestamp > deliveryDeadline and the escrow is still CREATED. Refunds the full amount back to the buyer. No fee.

escalateIfExpired

WRITE function escalateIfExpired(uint256 id)

Callable by anyone after the review deadline when state is DELIVERED. Pays the seller as if the buyer had confirmed. The fee still applies.

Use this to clear escrows where the buyer went offline. Useful in automated seller agents: watch Delivered, then schedule this call for reviewDeadline + 1 min with a gas-sponsored relay.

Events

Everything state-changing emits an event. Use these as webhooks-by-event-log — there is no HTTP webhook delivery.

event EscrowCreated(uint256 indexed id, address indexed buyer, address indexed seller, uint256 amount, uint64 deliveryDeadline, string verificationURI);
event Delivered(uint256 indexed id, bytes32 deliveryHash, uint64 reviewDeadline);
event Released(uint256 indexed id, uint256 toSeller, uint256 fee);
event Disputed(uint256 indexed id, address by, string reason);
event Cancelled(uint256 indexed id);
event Resolved(uint256 indexed id, uint256 toBuyer, uint256 toSeller, uint256 feePaid, bytes32 verdictHash);

Pattern: spin up a log subscription against EscrowV1 filtered by your buyer or seller address (for EscrowCreated) or by escrow ID (for everything else). See ethers.js ContractEvent.

The arbiter

EscrowV1 designates a single arbiter address at deploy time. When an escrow enters DISPUTED, only that address can call resolve(id, toBuyer, toSeller, verdictHash) to split the funds.

The arbiter is currently operated by Arbitova. The arbiter is not custodial — it cannot move funds to itself, only between the original buyer and seller. toBuyer + toSeller must equal amount at the solidity layer.

The verdictHash is a pointer to off-chain reasoning (IPFS, signed JSON, anything). It is emitted in Resolved so any observer can fetch the verdict and check it matches.

Long-term plan: open the arbiter role up to third parties per-escrow. For now, if you need a different arbiter, talk to us.
